## 2. Project Overview

- **Backend:** Spring Boot 3.5.0 (Java 17), Maven. MySQL via Spring Data JPA + Flyway (`ddl-auto=validate`). Spring Security (HTTP Basic, stateless), Actuator. Outbound integrations: MangaDex (read) and Twilio (SMS).

- **Config:** environment-variable driven (`${VAR}` placeholders) with a tracked `.env.example` template. Production overrides in `application-prod.properties` (fail-closed).
- **Deployment:** Railway (per `application-prod.properties` and the prod `apiUrl`).
- **Auth/session:** HTTP Basic, BCrypt-hashed in-memory user, stateless (no server sessions/cookies). Frontend holds the Basic token in `sessionStorage`.

## 4. Findings

### F1 — `.gitignore` did not ignore all `.env` variants
- **Severity:** Low
- **Location:** `/.gitignore`
- **Evidence:** Root ignored only `.env`, `.env.local`, `.env.*.local`; `.env.production` / `.env.staging` matched no rule (root, `backend/`, or `frontend/`).
- **Why it matters:** A real env file created with a non-`.local` suffix (e.g. `.env.production`) could be committed and published by accident — the most common way secrets leak.
- **Recommended fix:** Ignore every `.env` variant while keeping the committed template (`.env.*` + `!.env.example`).
- **Auto-fix status:** Fixed
- **Secret/value redacted:** N/A (no secret involved)
- **Confidence:** High

### F2 — Moderate DoS vulnerability in transitive `qs` dependency
- **Severity:** Low (moderate advisory; not on a clearly-reachable path for this app)
- **Location:** `frontend/node_modules/qs` (transitive; reported by `npm audit`)
- **Evidence:** `qs` 6.11.1–6.15.1 — `GHSA-q8mj-m7cp-5q26`, a remotely-triggerable `TypeError` in `qs.stringify` for null/undefined entries in comma-format arrays with `encodeValuesOnly`. 1 moderate finding total.
- **Why it matters:** Potential DoS where the vulnerable code path is exercised. This is a transitive build/tooling dependency of an SPA.
- **Recommended fix:** `cd frontend && npm audit fix`, then re-run `npm run test:ci` and `npm run build` to confirm no regression.
- **Auto-fix status:** Skipped — dependency changes are out of scope for automatic fixes (risk of behavior/lockfile drift); left as a one-command manual step.
- **Secret/value redacted:** N/A
- **Confidence:** High

### F3 — Phone numbers (PII) written to application logs
- **Severity:** Low
- **Location:** `backend/.../notification/TwilioNotificationService.java:53`; `backend/.../user/DataInitializer.java:25`
- **Evidence:** `log.info("SMS sent to {} ...", user.getPhoneNumber(), ...)` and `log.info("Created default user: {} ({})", name, phone)` emit full phone numbers.
- **Why it matters:** Phone numbers are personal data; emitting them in plaintext logs widens their exposure (log aggregation, shared dashboards).
- **Recommended fix:** Mask in logs (e.g. last 4 digits) or log a user id instead. Left manual because the masking format is a product decision and existing tests assert on these log paths.
- **Auto-fix status:** Manual action required
- **Secret/value redacted:** No phone values are reproduced here.
- **Confidence:** High

### F4 — Basic-auth token stored in `sessionStorage`
- **Severity:** Info
- **Location:** `frontend/src/app/auth.service.ts:7,15-16`
- **Evidence:** `sessionStorage.setItem('auth.basic', 'Basic ' + btoa(user:pass))`.
- **Why it matters:** Any stored credential is readable by injected scripts (XSS). `sessionStorage` (per-tab, cleared on close) is a reasonable choice for a Basic-auth SPA, and the interceptor logs out on 401, but the base64 token is reversible to the password.
- **Recommended fix:** Acceptable as-is for a single-user personal app. For hardening, move to a token-based scheme (short-lived bearer) so the long-lived password isn't client-resident. No change made — auth scheme changes are out of scope.
- **Auto-fix status:** Manual action required (by design; no action strictly needed)
- **Secret/value redacted:** N/A
- **Confidence:** Medium

### F5 — `anyRequest().permitAll()` after the `/api/**` matcher
- **Severity:** Info
- **Location:** `backend/src/main/java/com/mangatrack/SecurityConfig.java:68`
- **Evidence:** Non-`/api`, non-health paths are public.
- **Why it matters:** Usually a smell, but benign here: this is an API-only backend (the SPA is served separately), and non-health actuator endpoints are not exposed (`management.endpoints.web.exposure.include=health`), so there is no sensitive surface to reach.
- **Recommended fix:** Optionally switch the catch-all to `denyAll()` as defense-in-depth. No change made (judgment call; could affect error dispatch).
- **Auto-fix status:** Skipped (Info)
- **Secret/value redacted:** N/A
- **Confidence:** Medium

## 7. Authentication and Authorization Review

- HTTP Basic over TLS (prod `apiUrl` is HTTPS). Passwords **BCrypt-hashed** (`BCryptPasswordEncoder`).
- **Fails closed:** `SecurityConfig` throws on blank username/password; `application-prod.properties` provides no auth defaults. Dev fallback (`dev`/`local-dev-only`) is documented as local-only and overridden in prod. Covered by `SecurityConfigCredentialValidationTest`.
- Stateless sessions; `/api/**` requires authentication; OPTIONS and `/actuator/health` are public by design; unauthenticated requests get a clean 401 (no advice body).
- CSRF disabled — appropriate for a stateless, cookieless Basic-auth API.
- `Referrer-Policy: no-referrer` set. No automatic rewrites made (auth is out of scope).
