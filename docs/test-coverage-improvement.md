## 3. Highest-Value Coverage Gaps

### Gap G — `UserController` subscribe endpoint (`POST /api/users/{userId}/subscriptions`)

- **Location/files:** `backend/src/main/java/com/mangatrack/user/UserController.java`, `backend/src/test/java/com/mangatrack/user/UserControllerTest.java`
- **Why it matters:** This is the core subscription business rule. The endpoint must: 404 when the user doesn't exist, 404 when the manga doesn't exist, 409 when the user is already subscribed (a duplicate subscription would mean duplicate SMS for every new chapter), and on the happy path persist a `Subscription(userId, mangaId)` and return 201. The service-layer `SubscriptionServiceTest` only covers `autoSubscribeDefaultUser` — a different code path that does not touch this controller logic.
- **Existing tests:** Only `subscribe_missingMangaId_returns400` (body validation). None of the 404/409/201 branches are exercised.
- **Missing cases:** unknown user → 404; known user + unknown manga → 404; known user + manga + existing subscription → 409; known user + manga + no existing subscription → 201 persisting the right `(userId, mangaId)`.
- **Suggested tests:** extend `UserControllerTest` with four `subscribe_*` slice tests stubbing `userRepository.existsById`, `mangaRepository.existsById`, and `subscriptionRepository.existsByUserIdAndMangaId`/`save`.
- **Risk level:** Low (WebMvcTest slice, no behavior change).
- **Suggested validation:** `./mvnw test -Dtest=UserControllerTest`
- **Status:** Implemented

### Gap H — `UserController` list-subscriptions + unsubscribe (`GET`/`DELETE .../subscriptions`)

- **Location/files:** `UserController.java`, `UserControllerTest.java`
- **Why it matters:** `getSubscriptions` must 404 for an unknown user (rather than silently returning `[]`, which would hide a bad client request) and otherwise return the user's subscriptions as DTOs. `unsubscribe` is deliberately idempotent — deleting an existing subscription returns 204, and deleting a non-existent one is a no-op 204 (no 404, no crash). Neither endpoint is tested.
- **Existing tests:** none for these two endpoints.
- **Missing cases:** getSubscriptions unknown user → 404; getSubscriptions known user → 200 with DTO list; unsubscribe existing → 204 + `delete` called; unsubscribe absent → 204 + `delete` never called.
- **Suggested tests:** extend `UserControllerTest` with `getSubscriptions_*` and `unsubscribe_*` slice tests.
- **Risk level:** Low.
- **Suggested validation:** `./mvnw test -Dtest=UserControllerTest`
- **Status:** Implemented

### Gap I — `UserController.create` duplicate phone-number conflict (409)

- **Location/files:** `UserController.java`, `UserControllerTest.java`
- **Why it matters:** `create` wraps `userRepository.save` in `try/catch (DataIntegrityViolationException)` and maps the unique phone-number violation to a 409 with `A user with that phone number already exists`. This mirrors the already-tested `MangaController` duplicate-title 409. Without a test, a regression (swallowing the exception, leaking a 500, or wrong message) would go unnoticed.
- **Existing tests:** `create_validUser...`, `create_blankName...`, `create_invalidPhoneFormat...` — no conflict path.
- **Missing cases:** `save` throws `DataIntegrityViolationException` → 409 with `Conflict` envelope and the phone-number message.
- **Suggested tests:** one `create_duplicatePhone_returns409` slice test.
- **Risk level:** Low.
- **Suggested validation:** `./mvnw test -Dtest=UserControllerTest`
- **Status:** Implemented

### Gap J — `MangaController.add` optional-field persistence + auto-subscribe on success

- **Location/files:** `MangaController.java`, `MangaControllerTest.java`
- **Why it matters:** `add` copies three optional request fields onto the new entity (`mangadexId`, `coverUrl`, `noSource`) and, on a successful save, calls `subscriptionService.autoSubscribeDefaultUser(manga)`. The existing happy-path test sends only `{title, noSource:false}` and asserts the response title, so: (1) none of the optional fields are verified to reach the persisted entity — dropping `mangadexId` would silently disable chapter checks for manga added from a MangaDex search; and (2) the auto-subscribe call is only asserted *negatively* (in the 409 test) — no test proves it is invoked on success, so removing it would silently stop the default user's notifications.
- **Existing tests:** `add_validTitle_returns201WithDto` (title only); `add_duplicateTitle...` asserts auto-subscribe is *not* called on failure.
- **Missing cases:** add with `mangadexId` + `coverUrl` + `noSource:true` → saved entity carries all three (captured via `ArgumentCaptor<Manga>`) and `autoSubscribeDefaultUser` is invoked with the saved manga.
- **Suggested tests:** one `add_withOptionalFields_persistsThemAndAutoSubscribes` slice test.
- **Risk level:** Low.
- **Suggested validation:** `./mvnw test -Dtest=MangaControllerTest`
- **Status:** Implemented

### Gap K — `MangaDexService.extractTitle` non-English fallback chain

- **Location/files:** `MangaDexService.java`, `MangaDexServiceTest.java`
- **Why it matters:** MangaDex titles are a language-keyed map and frequently lack an `en` key for non-English series. `extractTitle` resolves in order: `title.en` → first `altTitles` entry with `en` → first value of the `title` map → `"Unknown"`. Only the `title.en` happy path is covered. A regression in this chain would mislabel titles in search results — directly user-visible — and the `"Unknown"` floor protects against an NPE/blank title when MangaDex returns no usable name.
- **Existing tests:** `searchManga_happyPath` (uses `title.en`).
- **Missing cases:** no `en` in `title` but an `altTitles` `en` → that alt; no `en` anywhere but a non-empty `title` map → first value; empty/absent title and alt titles → `"Unknown"`.
- **Suggested tests:** three `searchManga_*` slice tests driving `MockRestServiceServer` with the respective JSON shapes.
- **Risk level:** Low (uses the file's existing `MockRestServiceServer` harness).
- **Suggested validation:** `./mvnw test -Dtest=MangaDexServiceTest`
- **Status:** Implemented

### Gap L — `MangaCheckOrchestrator.tryStartManualCheckAll` executor-rejection reset

- **Location/files:** `MangaCheckOrchestrator.java`, `MangaCheckOrchestratorTest.java`
- **Why it matters:** After the `compareAndSet(false, true)` claims the run flag, the task submission is wrapped in `try { executor.execute(...) } catch (RuntimeException e) { manualRunInProgress.set(false); throw e; }`. If the executor rejects the task synchronously (`RejectedExecutionException` — pool saturated or shut down), the flag **must** be reset, otherwise it stays stuck `true` and every future manual check is permanently rejected with a 409. This compensating reset is untested; the existing tests use only inline and no-op executors that never throw.
- **Existing tests:** inline/no-op executor scenarios; none where `execute` throws.
- **Missing cases:** `executor.execute` throws → exception propagates AND `isManualRunInProgress()` is back to `false` AND a subsequent `tryStartManualCheckAll` can claim the slot.
- **Suggested tests:** one test with a throwing executor asserting the rethrow and the reset.
- **Risk level:** Low.
- **Suggested validation:** `./mvnw test -Dtest=MangaCheckOrchestratorTest`
- **Status:** Implemented
