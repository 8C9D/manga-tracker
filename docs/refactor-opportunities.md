## 1. Repository Overview

Manga Tracker is a two-tier app:

- **`backend/`** — Spring Boot 3.5 (Java 17), Spring Data JPA + MySQL (Flyway
  migrations), Spring Security (HTTP Basic), Actuator. Packages are cleanly
  split by domain: `manga`, `user`, `notification`, plus a small root package
  for cross-cutting web concerns (filters, CORS, security, global error
  handling). External integration with the MangaDex API and Twilio SMS.
- **`frontend/`** — Angular 21 (standalone components, signals), Vitest tests.
  A single feature surface: login, manga list, search/add. State lives in a
  `MangaAppFacade`; HTTP in `MangaService`/`AuthService`; pure helpers in
  `manga-utils.ts` and `http-error.ts`.

Build/test tooling: Maven wrapper (`./mvnw`) for the backend, npm + Angular CLI
for the frontend. `.mise.toml` pins toolchain versions.

## 3. Highest-Value Refactor Opportunities

### 3.1 Extract a shared validation-error body builder in `GlobalExceptionHandler`

- **Location/files:** `backend/.../GlobalExceptionHandler.java`
- **Problem:** `handleMethodArgumentNotValid` and `handleConstraintViolation`
  both build a `LinkedHashMap` of field errors, call
  `baseBody(HttpStatus.BAD_REQUEST, "Validation failed", request)`, then
  `body.put("fieldErrors", fieldErrors)`. The body-assembly tail was duplicated.
- **Why it matters:** The two handlers must keep an identical response shape
  (the frontend `http-error.ts` and tests depend on it). That contract was
  enforced by copy-paste in two places.
- **Suggested refactor:** Add `private static Map<String,Object>
  validationBody(Map<String,String> fieldErrors, WebRequest request)` and call
  it from both handlers.
- **Risk level:** Low
- **Expected benefit:** Single source of truth for the validation envelope.
- **Suggested validation:** `./mvnw -Dtest=GlobalExceptionHandlerTest test`,
  then full `./mvnw test`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented (commit `aedb1f7`).

### 3.2 Extract the repeated try/catch-log-return-empty wrapper in `MangaDexService`

- **Location/files:** `backend/.../manga/MangaDexService.java`
- **Problem:** `searchManga`, `fetchCoverUrl`, and `fetchLatestChapter` each
  wrapped their body in `try { ... } catch (Exception e) { log.warn(...);
  return <empty>; }`. The catch/log/return-default structure was triplicated.
- **Why it matters:** The "never let a MangaDex failure propagate; log and
  return empty" policy is the contract callers rely on. Centralizing it makes
  the policy explicit and impossible to diverge.
- **Suggested refactor:** A generic `private <T> T callOrEmpty(String label,
  String id, T fallback, Supplier<T> body)` helper; each public method becomes a
  thin call into it.
- **Risk level:** Low–Medium (external-integration path), but well covered.
- **Expected benefit:** Removes ~12 lines of duplicated error handling; one
  place to change the degradation policy/logging.
- **Suggested validation:** `./mvnw -Dtest=MangaDexServiceTest test`, then full
  `./mvnw test`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Implemented (commit `df98133`).

### 3.3 Extract the repeated `map(() => void 0)` response mapping in `MangaService`

- **Location/files:** `frontend/src/app/manga.service.ts`
- **Problem:** `checkAll`, `removeAll`, and `remove` all do
  `{ observe: 'response' }).pipe(map(() => void 0))` to turn a bodiless HTTP
  response into `Observable<void>`.
- **Why it matters:** Minor duplication; mostly a readability nicety.
- **Suggested refactor:** A small private `toVoid` helper, or leave inline.
- **Risk level:** Low
- **Expected benefit:** Marginal. Borderline over-abstraction.
- **Suggested validation:** `npm run test:ci`.
- **Dependency ordering:** Independent.
- **Autopilot status:** Skipped (see §6 — kept inline; idiomatic and clear).

## 6. Things Not Worth Refactoring Yet

- **`MangaService` `map(() => void 0)` (×3):** `observe: 'response'` +
  `map(() => void 0)` is the idiomatic Angular way to model a bodiless call as
  `Observable<void>`; a helper saves no real complexity and adds indirection.
- **`MangaAppFacade.confirmAdd` / `addNoSource`:** ~80% similar but reset
  different signals (`searchResults` vs `showAddAnyway`). Explicit is clearer
  than a parameterized merge here.
- **`MangaController.add` / `UserController.create` 409 mapping:** both translate
  `DataIntegrityViolationException` → 409, but with entity-specific messages in
  separate bounded contexts. Sharing would couple them for little gain.
- **Duplicated "Cannot reach the server…" string** in `login.component.ts` and
  `http-error.ts`: login deliberately can't route through `describeHttpError`
  (which returns `null` for 401), and the two contexts may legitimately diverge.
  Re-checked 2026-05-29; still not worth a shared constant.
- **`SecurityConfig` import grouping:** one import is mildly out of alphabetical
  order. Cosmetic only, not enforced by any linter, zero behavior value — left
  as-is to avoid pure churn.
- **`MangaAppFacade.sortedMangaList` re-computing `chaptersBehind` per
  comparison:** a micro-optimization over a tiny user-owned list; no observable
  effect, not worth the churn.
